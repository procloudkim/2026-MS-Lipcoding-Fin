# 02. 아키텍처 · 와이어프레임

> 시스템이 **어떻게 생겼고**, 화면 컴포넌트가 **어떻게 배치**되는지.

---

## 1. 시스템 아키텍처 (컴포넌트 다이어그램)

```mermaid
flowchart TB
    subgraph Browser["🖥️ 브라우저 (Client)"]
        UI["index.html<br/>입력 패널 · 결정 무대"]
        APP["app.js<br/>상태관리 · fetch · 취소 · 정렬조정"]
        LS[("localStorage<br/>마지막 결과")]
        UI <--> APP
        APP <--> LS
    end

    subgraph Azure["☁️ Azure App Service (Linux · B1)"]
        subgraph Node["Node 런타임 (server.js)"]
            HELMET["helmet 보안헤더"]
            STATIC["정적 서빙 public/"]
            API["REST API<br/>/api/health · /api/plan · /api/assist"]
            CORE["lib.js 도메인 코어<br/>프롬프트·파서·정렬·폴백"]
            HELMET --> STATIC
            HELMET --> API
            API --> CORE
        end
        ENV["App Settings<br/>COPILOT_GH_TOKEN(시크릿)"]
    end

    subgraph AI["🤖 Copilot 에이전트"]
        SDK["@github/copilot-sdk<br/>CopilotClient 세션"]
        RT["@github/copilot 런타임<br/>(자식 프로세스)"]
        SDK <-->|JSON-RPC| RT
    end

    APP -->|HTTPS POST JSON| API
    API -->|토큰 인증 세션| SDK
    ENV -.토큰 주입.-> SDK
    CORE -.SDK 실패 시 폴백.-> API
```

**핵심 설계 포인트**
- **단일 서비스**: 프런트(정적)와 API가 한 Node 프로세스에 공존 → 배포 단위 1개.
- **AI는 곁가지**: `/api/plan`은 우선 SDK를 시도하고, 실패하면 `lib.js`의 결정적 폴백으로 자연 전환 → **AI가 죽어도 앱은 산다**.
- **시크릿 격리**: 토큰은 코드가 아니라 App Settings에서 환경변수로 주입.

## 3. 반응형 거동

| 폭 | 레이아웃 |
| --- | --- |
| **≥ 960px** (데스크톱) | 좌(입력)·우(결정 무대) 2단. 입력 패널 `sticky`. |
| **560–960px** (태블릿) | 1단 세로 스택. 입력 → 결과 순. |
| **< 560px** (모바일) | 결정 카드가 `rank → chip → body → 컨트롤` 세로 재배치. 버튼 풀폭. |

## 4. 화면 상태 머신 (UX State)

```mermaid
stateDiagram-v2
    [*] --> empty: 첫 진입
    empty --> loading: "오토파일럿 실행"
    loading --> result: 200 응답(파싱 성공)
    loading --> error: 네트워크/서버 오류
    loading --> empty: 사용자 "취소"(AbortError)
    error --> loading: "다시 시도"
    result --> loading: 재실행
    result --> result: 정렬 조정(위로/지금으로) · 초안 생성
    note right of result
        localStorage 저장 →
        새로고침 시 result 복원
    end note
```

> 4개 상태(empty·loading·error·result)는 `[hidden]` 속성으로 **상호 배타** 렌더링.
