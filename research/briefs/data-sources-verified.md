# 추천 데이터 출처 검증 — "어디서 가져오나" (크롤링 vs 공식 Open API)

> 질문: 제품을 추천(또는 안전 판단)하려면 데이터를 **어디서 크롤링/수집**하나?
> 결론 먼저: **크롤링이 아니라 정부 공식 Open API**가 정답이다. 한국 영유아/공산품 안전·리콜·인증을
> 프로그래밍으로 주는 **실재하는 무료 API**가 있고, 이게 책임/정직성/법적 안전 모두에서 우월하다.
> Last verified: 2026-06-20 (아래 각 항목 근거 URL/실호출 표기).

---

## 0. 핵심 통찰 (먼저 읽기)

공식 API는 **"이건 사지 마라"(negative signal)는 강하게** 줄 수 있지만, **"이걸 사라"(positive 추천)는 못 준다.**
- 리콜/회수 데이터 = **피해야 할 제품**을 제품명·모델명·바코드로 조회 가능 → `avoid[]` / 안전성 경고에 직결.
- KC 인증 데이터 = **인증 유무 확인** → "인증된 것을 고르라"는 *기준*은 주지만, *특정 브랜드 추천*은 아님.
- 즉 10라운드 토론의 결정(**브랜드 추천 금지 · 근거 없으면 INSUFFICIENT**)과 정확히 맞물린다:
  데이터는 **추천 사양·피할 신호·리콜 검증**을 뒷받침하고, 브랜드 추천은 여전히 금지.

### 소비자24 리콜 API 상세(개발가이드 기준)
- URL: `https://www.consumer.go.kr/openapi/recall/contents/index.do?serviceKey={키}&pageNo=1&cntPerPage=10&cntntsId={메뉴ID}`
- 선택 파라미터: `productNm`(제품명) · `bsnmNm`(사업자) · `modlNmInfo`(모델명) · `stdBrcd`(바코드) · `recallProcssInfo`(인증번호)
- 메뉴 ID(영유아 관련): `0101 공산품` · `0208 위생용품` · `0206 화장품` · `0201 식품` · `0205 의약외품` · `0501 해외리콜`
- 응답 필드: `productNm·makr·modlNmInfo·stdBrcd·shrtcomCn(결함내용)·injryCauseResult(위해원인)·cnsmrGhvrTips(소비자 행동요령)·infoOriginUrl(출처)·recallImgUrls`
- → **"이 젖병/쪽쪽이가 리콜됐나?"를 제품명·모델명·바코드로 조회**하는 정확한 수단.

| 출처 | 확인 방법 | 결과 |

## 3. 크롤링은 왜 답이 아닌가 (쿠팡/네이버/제조사 사이트)

| 항목 | 문제 |
| --- | --- |
| 이커머스(쿠팡·네이버쇼핑) | 이용약관·robots.txt 위반 소지. 상품 DB 무단수집은 **DB권/부정경쟁 분쟁 위험**(예: 잡코리아 v 사람인, 네이버 v 크롤러 류 판례 경향). 상업 서비스에 부적합 |
| 제조사 공식 사이트 | 구조 비표준·수시 변경 → 시험성적서/인증 파싱 **신뢰도·유지보수 최악**. 깨지기 쉬움 |
| 가격/판매량 | 안전과 무관하고 편향(싼 게 위험) 조장 → 본 앱 가치와 충돌(토론 R5) |
| 데이터 정합성 | 크롤링은 출처·시점 보증 불가 → 정직성(R3) 위반 위험 |

→ 결론: **크롤링 대신 정부 공식 Open API**. 출처가 공신력 그 자체이고, 약관상 재사용이 명확하며(대부분 data.go.kr
이용허락범위 "제한 없음", 단 데이터셋별 확인), JSON으로 안정적.

## 4. 라이선스/약관 (확인 필요 주의)

- **data.go.kr**: 데이터셋별 "이용허락범위" 표기. 다수가 제한 없음(상업·2차가공 가능)이나 **데이터셋마다 개별 확인 필수**.
- **소비자24**: serviceKey(승인번호) 발급 후 사용. 출처표시 권장.
- **openFDA**: terms/license 공개(고지 의무, 의료 판단 의존 금지 disclaimer).
- **CPSC**: 미국 연방 공공데이터(일반적으로 자유 이용, 출처표시 권장).

## 5. 본 앱에 맞는 현실적 데이터 전략 (Top 3, 우선순위)

1. **소비자24 통합 리콜 API**(공산품·위생용품·화장품·해외리콜) — *제품명/모델명/바코드 리콜 조회*.
   가장 직접적으로 "피해야 할 신호(avoid)"와 "리콜 검증"을 데이터로 채움.
2. **국가기술표준원 제품안전인증·리콜 API**(15116894) — *어린이제품 KC 인증 + 리콜*.
   "KC 인증 확인" 체크리스트를 **실데이터**로 승격(현재는 라벨만).
3. **식약처 화장품/식품 회수 API** — 로션·물티슈·이유식 등 *회수 이력*.

해외(CPSC/openFDA)는 4순위 보강. 가격/이커머스 크롤링은 **DEFER/금지**.

### 권장 아키텍처(점진 도입, B1 정공법)
```
사용자 입력(제품명/모델/우려)
  -> 제품군 판정(카테고리)
  -> [Retriever] 소비자24 리콜 + KC 인증 + (해외 CPSC) 병렬 조회   ← 실데이터
  -> 근거 chunk(리콜 유무·인증 유무·출처URL)
  -> Copilot SDK: 근거를 인용해 판정/추천 사양/피할 신호 생성(근거 없으면 INSUFFICIENT)
  -> 화면: 리콜 경고 배지 + 출처 링크 + 추천 사양/피할 것
```
- 키는 App Setting/.env로만(커밋 금지). 호출 실패 시 기존 fallback로 graceful.
- 이게 B1(실데이터 미연동)·B3(환각)을 동시에 해소하는 정공법.

## 6. 정직성 한계(반드시 명시)

- 공식 API도 **"리콜 안 됨 = 안전 보증"이 아니다.** (등록 누락·신제품·해외 미반영 가능)
- 추천은 여전히 **사양/기준 수준**(브랜드 금지). 데이터는 *경고·검증·기준*을 강화할 뿐 "이 제품을 사라"를 만들지 않는다.
- 리콜 조회 0건 → "현재 등록된 리콜 없음(보증 아님)"으로 표현, INSUFFICIENT 규율 유지.
